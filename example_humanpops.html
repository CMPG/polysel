Detection of polygenic selection in genesets (polysel)

Josephine Daub

Wednesday, March 16, 2016

Introduction

Here we describe step by step how to use the gene set enrichment pipeline, as it was applied to detect pathways that show signals of adaptation in recent human history.

The dataset and testing procedure is described in:

Daub, J.T. et al. (2013). Evidence for Polygenic Adaptation to Pathogens in the Human Genome. Mol Biol Evol 30, 15441558.

All code is written in the programming language R (https://www.r-project.org/). This example is run with R version 3.1.0 (2014-04-10). Part of the code can be run on a desktop PC, but another part -especially the construction of the empirical p-value null distribution- should run on a cluster. Instructions on how to submit the code on a cluster are described below.

We advice to use the following directory structure, both on your PC and on a cluster:

  • polysel
    • R - R files with all functions needed to run the pipeline
    • data - Input text files with gene and gene set information, but also the R data object files that are created during data preparation. Each project has its own subfolder
      • altitude
      • humanpops
      • primates
        • homininae
        • hominidae
        • hominoidae
        • catarrhini
    • empfdr - Output files from the creation of the p-value null distribution (see below)
    • log - Location of error and output log files, mainly used on the cluster
    • script - Script files to run the pipeline on a cluster
    • results - Results of the gene set enrichment test, namely a list of gene sets with their scores and p and q values

Data preparation and inspection

In order to run the functions of the pipeline, make sure that the packages qvalue and Matrix are installed. You can install them with:

install.packages("qvalue")
install.packages("Matrix")

Next, define the project name and set the path to input and output files:

# set project variables
project.name<-"humanpops"
data.path<-file.path("./data/humanpops")
# set paths
code.path<-"./R"
empfdr.path<-"./empfdr"
results.path<-"./results"

The file polysel.R contains all the functions to run the pathway enrichment pipeline.

source(file.path(code.path,'polysel.R'))

Read in the input tables. We need three files:

  1. A file with gene set information (here ‘SetInfo.txt’) that contains at least the fields ‘setID’ and ‘setName’.
  2. A file with gene (object) information (here ‘ObjInfo.txt’) that contains at least the fields ‘objID’, ‘objStat’ and ‘objName’.
  3. A file linking gene sets with genes (here ‘SetObj.txt’) that contains at least the fields ‘setID’ and ‘objID’.

Make sure that all these fields have a header, and that they are tab separated.

We set a minimum set size to exclude too small pathways for which we usually don’t have enough power to detect signals of selection.

With the SUMSTAT pathway enrichment test, we test if pathways (gene sets) show a shift in the distribution of their gene scores. Then the SUMSTAT score (the sum of the gene scores) is compared with a null distribution of SUMSTAT scores. The creation of such null distribution depends on several factors, they will be explained in the next paragraphs.

Check for bias

The gene score can be correlated to gene properties such as gene length or SNP density. For example, long genes will have on average more mutations and for this reason you will have more power to detect signals in long genes compared to small genes. If this is the case, we should correct for this in order to avoid biased test results. To investigate possible biases, you can plot the gene scores against the gene length, number of SNPs per gene or other gene properties. In the humanpops dataset we investigate the correlation between gene score and total SNP density (SNPcount).

flds<-c("SNPcount")
options(digits=2)
for (fld in flds){
  cat("Correlation between objStat and ", fld, ": ", sep="")
  ct<-(cor.test(obj.info$objStat,obj.info[[fld]]))
  cat("p-value: ", ct$p.value, " estimate: ", ct$estimate, 
      " [", ct$conf.int[1],
      ", ", ct$conf.int[2],"]\n",sep="")
  PlotStatField(obj.info, fld=fld, ylab=fld, xlab="objStat", 
                logaxis="y", show.bins=F)
}
> Correlation between objStat and SNPcount: p-value: 0 estimate: 0.35 [0.33, 0.36]

If there is a bias, you can correct for that by creating bins with genes that have similar values of the gene property. Then you can rescale the gene scores within each bin, or create a null distribution of random gene sets that follow the same ‘bin distribution’ as the set of interest.

Here, we find a correlation between gene score and SNP density (SNPcount). We therefore assign the genes to SNP density bins:

obj.info<-AssignBins(obj.info,fld="SNPcount")
obj.info<-obj.info[order(obj.info$objID),]

We can visualize the bin boundaries (setting show.bins=T):

options(digits=2)
for (fld in flds){
    PlotStatField(obj.info, fld=fld, ylab=fld, xlab="objStat",
                logaxis="y", show.bins=T)}

As the gene scores are rather symmetrically distributed per bin, we can rescale the objStat score per bin.

obj.info<-RescaleBins(obj.info)

The correlation between gene score and SNP density is now almost gone:

options(digits=2)
for (fld in flds){
  cat("Correlation between objStat and ", fld, ": ", sep="")
  ct<-(cor.test(obj.info$objStat,obj.info[[fld]]))
  cat("p-value: ", ct$p.value, " estimate: ", ct$estimate, 
      " [", ct$conf.int[1],
      ", ", ct$conf.int[2],"]\n",sep="")
  PlotStatField(obj.info, fld=fld, ylab=fld, 
                xlab="objStat", logaxis="y", show.bins=T)
}

Check if gene scores are normally distributed

If you do not use bins and if sum(objStat) approximates a normal distribution, there is no need to create a null distribution. We can then use properties of all genes (namely the mean and variance of their scores), to infer the null distribution, which is much faster than creating one.

However, we can only do this if for small and large random gene sets we indeed find a Gaussian distribution of the SUMSTAT scores.

for (sz in c(10,50,250)){
  CheckStatDistribution(obj.info,setsize=sz,
                        n.rand = 100000, 
                        xlab="sum(objStat)")
}

In our case we clearly see that even for small sets the sum(objStat) values are approximating a normal distribution, so we can do without the creation of a null distribution.

After checking for biases and checking whether the null distribution is gaussian, and -if needed- correcting objStat for these biases, we create the data frame obj.stat which is derived from obj.info and consists of its fields objID, objStat and objBin. We save all the data objects so we can use them later on for testing.

obj.stat<-obj.info[,c("objID", "objStat", "objBin")]
save(set.info, obj.info, obj.stat, set.obj, 
     file=file.path(data.path, "polysel_objects.RData"))

Set parameters

In our case, as we can approximate the SUMSTAT scores of random sets with a normal distribution, we don’t have to create a null distribution by randomly sampling genes from the list of genes:

approx.null <- TRUE

Furthermore, we assigned genes to bins, but because we rescaled their score within bins, we don’t use the bins anymore:

use.bins <- FALSE

If we have to create a null distribution (approx.null=FALSE), we use a sequential random sampling method to speed up the construction of null distributions. In short, for each tested gene set, we create a small null distribution and calculate a temporary p-value. For those sets that are below a certain p-value threshold, we repeatedly increase their corresponding null distribution until a maximum number of randomizations is reached. In our case, since we don’t create a null distribution, this parameter will not be used.

seq.rnd.sampling <- FALSE

If we have to create a null distribution (approx.null=FALSE), we need to specify the number of random sets to create the null distribution to a sufficiently high value (> 400000 ). If you use sequential random sampling (seq.rnd.sampling = TRUE), you set the maximum number of randomizations. You can set to a lower value for nrand when you first want to test the function. In our case, this parameter will not be used and we set it therefore to 1.

nrand <- 1

Usually we are looking for gene sets with extremely high SUMSTAT scores, so we compare their scores to the higher tail of the null distribution.

test <- "highertail"

Use lowertail or twosided if you are searching for low SUMSTAT scores or extreme scores respectively.

We use the function qvalue from the R package qvalue to calculate the q-value with the p-values of the tests.

We set here the parameter pi0.method of this package to its default value. See ?qvalue for more information about this function.

qvalue.method="smoother"

Running gene set test without pruning

The enrichment pipeline will run two rounds of tests. The first round will test all gene sets and returns their p-value (and corresponding q-value). This first run usually doesn’t take a lot of time and can be done on a pc.

# Test sets
result<-EnrichmentAnalysis(set.info, set.obj, obj.stat,
                    nrand=nrand, approx.null=approx.null, 
                    seq.rnd.sampling=seq.rnd.sampling,
                    use.bins=use.bins, test=test,
                    do.pruning=FALSE, minsetsize=minsetsize,
                    project.txt=project.name, do.emp.fdr=FALSE,
                    qvalue.method=qvalue.method
                    )

set.scores.prepruning <- result$set.scores.prepruning

Running gene set test with pruning

In the second round the overlap is removed between gene sets in a process called pruning. In short, the genes from the highest scoring gene set will be removed from all lower ranked gene sets and these gene sets will be retested. This process is repeated until no large enough sets are left to be tested. This pruning step will take more time (about 30 minutes on a single pc) and will return a shorter list of pruned gene sets with their p-values.

# Test sets
result<-EnrichmentAnalysis(set.info, set.obj, obj.stat,
                    nrand=nrand, approx.null=approx.null, 
                    seq.rnd.sampling=seq.rnd.sampling,
                    use.bins=use.bins, test=test,
                    do.pruning=TRUE, minsetsize=minsetsize,
                    project.txt=project.name, do.emp.fdr=FALSE,
                    qvalue.method=qvalue.method
                    )

set.scores.postpruning <- result$set.scores.postpruning

Creating p-value null distribution to infer FDR empirically

The pruning test will result in a list of p-values biased toward lower p-values. We will therefore estimate the FDR using a empirical p-value distribution. For this we will repeatedly shuffle the gene scores and retest the gene sets (including pruning). In order to calculate an empirical FDR on the p-values that are the result of the test with pruning, we need to create a null-distribution of p-values. This is done by permuting (shuffling) the gene scores while keeping the gene set definition as is (set.obj). After each permutition the gene sets are tested with pruning and the resulting p-values are stored in a file. Usually around 200-300 of such randomizations are performed. On a single desktop this could take up to 150 hours.

This part should best be run on a cluster.

On a cluster one could easily create and run one job per randomization reducing the computation time significantly. We provide an R script and example bash scripts to submit such jobs. Make sure which submission system your cluster is using and what is the policy to submit jobs on your cluster, as this differs per organization.

To test functionality on a single PC, one can run only a small number of randomizations.

# to see if everything works fine, you can first set a low number of runs to test the function
emp.fdr.nruns <- 3
for (r in seq(from=1, to=emp.fdr.nruns)){
# Test randomized sets
TestShuffledSets(set.info, set.obj, obj.stat, 
                 nrand=nrand, minsetsize=minsetsize, 
                 approx.null=approx.null, test=test,
                 seq.rnd.sampling=seq.rnd.sampling,
                 use.bins=use.bins, out.path=empfdr.path, 
                 runnr=r, project.txt=project.name)
}

Creating p-value null distribution on a CLUSTER

To create the p-value null distribution on a cluster, we provide an R file polysel_shufflesets_cluster.R that calls the TestShuffleSets function with commandline arguments. This file can be found in the ./R folder. You can then use the script polysel_shufflesets_cluster.sh that calls this R file (in ./script folder).

Make sure to copy the dataset (polysel_objects.RData) to the corresponding data directory (in this case ./data/humanpops)

The submission of a job that calls this script depends on the type of cluster submission system you use and the submission rules that apply in your organization. Here is an example how you can submit a job that calls the polysel_shufflesets_cluster.sh script on a SGE (Sun Grid Enginge) cluster. Make sure that you call this script from the project root directory (polysel in our example). More examples of these commands can be found in the file Examples_submit_on_cluster

projectname="humanpops"
for i in {1..200}
do
qsub -l h_cpu=2:00:00 -l h_vmem=4G -o ./log/shuffle${i}_${projectname}.out \
-e ./log/shuffle${i}_${projectname}.err -cwd -N shuffle${i}_${projectname} \
./script/polysel_shufflesets_cluster.sh $i $projectname
done

Calculation of empirical FDR

This part can be run on your desktop. On the cluster, you would usually do the test with and without pruning together with the calculation of the empirical FDR in one go. If you run the test on your desktop and if the p-value null distribution is created on cluster, then first move the corresponding files from the ./empfdr folder on the cluster to the same folder on your desktop (./results).

Run on a desktop:

# estimate m0, the proportion of true nulls

Run on a cluster:

To run the complete test (with and without pruning) on a cluster, you can use the script polysel_cluster.sh (located in ./script folder) that calls the R file polysel_cluster.R (located in ./R folder)

Again, the submission of a job that calls this script depends on the type of cluster submission system you use and the submission rules that apply in your organization. Here is an example how you can submit a job that calls the polysel_cluster.sh script on a SGE (Sun Grid Enginge) cluster. Make sure that you call this script from the project root directory (polysel in our example).

projectname="humanpops"
qsub -l h_cpu=5:00:00 -l h_vmem=4G -o ./log/${projectname}.out \
-e ./log/${projectname}.err -cwd -N ${projectname} \
./script/polysel_cluster.sh $projectname
